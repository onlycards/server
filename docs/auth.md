# Авторизация

Пользователь — это ID, имя, пароль и роль.
Пароль не может быть менее 8 символов. Доступные роли: ученик и преподаватель.

В БД пароль хранится в виде sha256 с ID пользователя в качестве соли.

Ученик может войти только в клиентское приложение. Преподаватель может войти
как в клиентское приложение, так и в панель администратора. Это два разных
фронтенда, так что им нужны независимые токены доступа.

Вход в приложение означает выдачу токена доступа в виде HttpOnly Cookie.
Любой авторизованный пользователь может сменить свой пароль.

> API не предоставляет регистрацию.
> Регистрация производится отдельно администратором СУБД.

## Middlewares

Все пути приложения разделены на блоки:

- `enter`, пути входа;
- `common`, общие пути.
- `client`, пути пользователя;
- `admin`, пути администратора;

В файле `/src/app.ts` каждому из этих блоков заданы
промежуточные обработчики (для `common` и `client` одинаковые).

Логика промежуточных обработчиков находится в директории `/src/auth`.

### Enter Middleware

Пути входа недоступны, если пользователь уже авторизован.
Если клиентское приложение позволило отправить запрос входа,
имея активную сессию, это проблема клиентского приложения
и она должна быть подсвечена.

При возникновении такой ситуации, на всякий случай сбрасываем сессию.

### Common/Client Middleware

В этих блоках обработчики ожидают авторизованные запросы от имени пользователей.
Соответственно, Middleware проверяет, чтобы в запросе был токен доступа
соответствующий активной сессии существующего пользователя.

Если сессия не обновлялась одни сутки,
то её срок жизни будет обновлён (сессия живёт неделю).

### Admin Middleware

Делаёт всё то же, что и Common/Client Middleware. Дополнительно проверяет,
чтобы сессия была выдана именно для панели администратора
и чтобы пользователь имел роль преподавателя.

Если же, например, админ перенесёт токен, выданный для панели администратора,
в клиентское приложение, то в этом нет ничего страшного.

## Пути входа

Находятся в директории `/src/routes/enter`.

Есть путь для авторизации в клиентском приложении
и путь для авторизации в панели администратора.

Условием входа является совпадение хэша пароля.
При входе создаётся сессия в БД, пользователь получает Cookie.

## Общие пути

Обработчики выхода из учётной записи и смены пароля являются общими
для клиентского приложения и панели администратора, так что объединены
в отдельный блок `/src/routes/common`.

При смене пароля сессия не сбрасывается.
